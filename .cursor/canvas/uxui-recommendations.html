01

Accesibilidad & Semántica

Idioma incorrecto en <html lang>
Crítica

El layout declara lang="en" pero todo el contenido está en español. Esto afecta lectores de pantalla, síntesis de voz y herramientas de traducción automática.

// layout.tsx <html lang="en"> // → Corrección <html lang="es">
Accesibilidad
SEO
<Link><button> — anidado inválido
Crítica

En MenuHeader.tsx, cada ítem de navegación envuelve un <button> dentro de un <Link> (que renderiza como <a>). Es HTML inválido: un elemento interactivo no puede contener otro.

// Patrón actual (inválido) <Link href="/shop"> <button className={getButtonClass('/shop')}> Tienda </button> </Link> // → Corrección: estilizar el Link directamente <Link href="/shop" className={getButtonClass('/shop')}> Tienda </Link>
Accesibilidad
HTML válido
hide-scrollbar en <body>
Alta

Ocultar el scrollbar del body elimina una señal visual esencial: el usuario no puede saber cuánto contenido hay debajo del fold. Empeora la experiencia en páginas largas (blog, tienda).

Recomendación: Mantener el scrollbar nativo visible. Si el objetivo es estética, usar un scrollbar personalizado con CSS scrollbar-width y scrollbar-color en lugar de ocultarlo.
Usabilidad
02

Navegación

Marquee como link en mobile — sin affordance
Alta

En MobileHeader.tsx, el título en marquee es el único link a Home en mobile. Los usuarios no saben que es clickeable. Un texto en movimiento que funciona como link viola principios de UX de descubribilidad.

Recomendación: Agregar un logo/logotipo estático o icono de Home separado con un tap target mínimo de 44×44px. El marquee puede seguir siendo decorativo.
Descubribilidad
Mobile UX
Carrito sin acceso directo en mobile
Alta

En mobile, el carrito está enterrado dentro del Dropdown. El badge de items requiere abrir el menú para verlo. Esto aumenta el abandono en el flujo de compra.

Recomendación: Agregar un ícono de carrito con badge siempre visible en el header mobile (fuera del dropdown), siguiendo el patrón estándar de e-commerce.
Conversión
Sin breadcrumbs en detalle de producto / post
Media

Las páginas /product/[slug] y /blog/[slug] no tienen breadcrumbs. El usuario no sabe dónde está dentro de la jerarquía y el botón "atrás" del browser es la única forma de volver.

Recomendación: Añadir breadcrumb simple: Inicio → Tienda → Nombre del libro. Mejora navegación y SEO (structured data).
Orientación
SEO
TopTicker ausente en mobile
Baja

El TopTicker está oculto en mobile (hidden md:flex). En desktop es el único elemento que muestra el nombre de la editorial prominentemente, pero en mobile no existe ese componente de branding.

Recomendación: Mantener algún elemento de identidad de marca siempre visible en mobile. El marquee actualmente sirve ese propósito, pero si se rediseña la navegación, evaluar incluir el nombre de la editorial en el header.
03

Tienda & Blog

Sidebar de Bio — bajo aprovechamiento del espacio
Alta

El sidebar ocupa 25vw (hasta 400px) y solo contiene una foto, el tagline "No es desierto. Es atropello." y un link a Instagram. Es una gran área de pantalla con muy poca información útil.

Recomendación: Enriquecer el sidebar con: descripción breve de la editorial, últimas publicaciones destacadas, novedades del blog, o un CTA hacia la tienda. Alternativamente, considerar moverlo a la Home page y hacer el sidebar colapsado por defecto.
Contexto
Tienda sin filtros ni ordenamiento
Alta

La página /shop muestra todos los productos en lista, ordenados solo inversamente por índice (editions.slice().reverse()). No hay filtros por precio, género, autor, ni ordenamiento.

Recomendación: Agregar al menos: (1) ordenamiento por precio ascendente/descendente y (2) filtro por disponibilidad. Con un catálogo creciente, la usabilidad cae drásticamente sin estas herramientas.
Conversión
Blog sin paginación
Media

Todos los posts se cargan de una sola vez en /blog/page.tsx (getPostsSorted()). A medida que crezca el catálogo, esto impactará fuertemente en el tiempo de carga y el consumo de datos.

Recomendación: Implementar paginación server-side o infinite scroll con cursor-based pagination en Contentful. Un límite inicial de 12–15 posts es razonable.
Performance
Escalabilidad
Tags sin conteo de artículos
Media

Los filtros por etiqueta no muestran cuántos artículos hay en cada categoría. El usuario no puede anticipar si una etiqueta tiene 1 o 20 artículos antes de hacer clic.

Recomendación: Mostrar el conteo junto a cada tag: Poesía (8). Si es costoso calcularlo en el servidor, cachear el resultado con revalidate.
Imágenes en escala de grises en el blog
Media

Todas las imágenes de PostItem tienen className="grayscale" aplicado. Si es una decisión de branding, es válida, pero reduce el engagement en grids visuales y puede hacer las tarjetas difíciles de diferenciar.

Recomendación: Considerar mostrar el color en hover (ya existe la transición de escala). Cambiar a grayscale group-hover:grayscale-0 transition-all duration-500 para dar un toque de descubrimiento visual.
dangerouslySetInnerHTML en reseña de producto
Baja

En ShopItem.tsx, la reseña se renderiza con dangerouslySetInnerHTML. Si el contenido viene de Contentful (rich text), evaluar usar el renderer oficial @contentful/rich-text-react-renderer para mayor seguridad y flexibilidad tipográfica.

// ShopItem.tsx — riesgo si el HTML no está sanitizado dangerouslySetInnerHTML={{ __html: review }} // → Alternativa más segura import { documentToReactComponents } from '@contentful/rich-text-react-renderer' documentToReactComponents(reviewRichText)
04

Performance & Pulido visual

Múltiples backdrop-blur simultáneos
Media

El header, los botones de nav, el sidebar y el dropdown usan todos backdrop-blur-lg al mismo tiempo. En dispositivos de gama media-baja, múltiples capas de blur activas generan jank visible.

Recomendación: Reducir el nivel de blur (backdrop-blur-sm) o limitarlo a 1–2 elementos por vista. El sidebar y el header raramente coexisten visualmente, por lo que el impacto puede ser manejable.
GPU pressure
Sin footer
Baja

La web no tiene footer. El contenido termina abruptamente. Para una editorial, el footer es un lugar canónico para: contacto, redes, derechos de autor, política de envío, y links secundarios.

Recomendación: Agregar un footer minimalista con: © año, links de navegación secundaria, Instagram, y email de contacto. Mantenerlo austero para no romper la identidad visual.
Opacity 80% en estado activo de nav
Baja

Los botones de navegación tienen opacity-80 como clase base incluso en estado activo. Esto reduce la legibilidad del texto cuando el elemento está seleccionado — el estado activo debería ser el de mayor contraste.

// baseClass actual "... opacity-80 ..." // → Separar opacity por estado isActive ? "... opacity-100 ..." : "... opacity-70 hover:opacity-100 ..."
